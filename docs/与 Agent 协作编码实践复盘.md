# 与 Agent 协作编码实践复盘

## 1. 文档目的

本文档基于本次“统一知识导入改造”协作过程，总结与 Agent 协作编码的有效方法、常见风险和可复用经验。

目标有三点：

- 沉淀一套适合本项目的 Agent 协作方式
- 对照外部最佳实践，确认本次做对了什么
- 归纳下次协作时可以更早做、做得更稳的事项

## 2. 本次协作的总体结论

本次协作整体是成功的，原因不是“Agent 自动完成了开发”，而是你把协作过程经营成了一个比较标准的软件工程流程：

- 先定义目标，再讨论方案
- 先写 spec，再写 plan，再进入实现
- 关键决策由人拍板，Agent 负责展开与落地
- 遇到运行时问题时，提供真实日志而不是只描述现象
- 最后要求文档沉淀、测试验证、分支合并与清理

这和主流 AI 编码最佳实践是高度一致的。外部资料普遍强调一件事：人应当保持主导，AI 负责加速、探索、生成和重复性工作，而不是替代判断。

## 3. 外部最佳实践摘要

结合 GitHub、OpenAI、Anthropic 的官方资料，可以把与 Agent 协作编码的最佳实践概括为 5 条。

### 3.1 人负责判断，Agent 负责放大效率

GitHub 官方文档明确指出，AI 编码助手能帮助开发者更快工作，但不应替代开发者的专业判断；用户应理解它的强项与边界，并始终对结果负责。

落到实践上就是：

- 让 Agent 做架构展开、样板代码、重构、测试补齐、文档沉淀
- 人来决定目标、范围、边界条件、风险取舍和最终验收

### 3.2 提示要具体，复杂任务要拆解

GitHub 和 OpenAI 都强调，复杂任务应分解，要求要具体，最好附带输入输出、约束和示例。

高质量协作提示通常应包含：

- 要解决的业务目标
- 当前代码或文件位置
- 明确约束
- 希望的产物
- 不希望采用的路径

### 3.3 上下文要充分，但要聚焦

GitHub 建议提供相关文件和上下文，并及时移除无关上下文；OpenAI 也强调要提供相关背景信息，让模型能基于真实上下文工作。

对工程协作来说，最有价值的上下文通常是：

- 现有代码路径
- 错误日志
- 运行环境
- 依赖限制
- 已做过的决策

### 3.4 一定要验证，不要把“生成”当“完成”

GitHub 官方把“检查 AI 的工作”列为核心原则，强调要理解建议代码、审查可维护性与安全性，并用自动化测试和工具验证结果。

这意味着：

- 代码生成不是完成
- 测试通过也不是全部完成
- 真实样本跑通、迁移跑通、环境跑通，才更接近“完成”

### 3.5 把项目规则写下来，让 Agent 每次都能复用

Anthropic 官方文档强调，应把编码标准、架构约束、检查清单等沉淀到项目级说明中，让 Agent 在每次会话开始时读取；同时也强调可以通过 instructions、skills、hooks、MCP 等手段持续强化协作质量。

这条对团队长期收益最大：

- 不要每轮都从零讲规则
- 把高频约束写进项目文档和仓库约定
- 让 Agent 复用稳定规则，而不是靠临时口头提醒

## 4. 这次你做对的地方

本次合作里，你有几件事做得非常对，而且确实提高了结果质量。

### 4.1 先做方案讨论，而不是直接让 Agent 改代码

你一开始没有直接要求“马上写”，而是先讨论：

- 目标是什么
- 方案一、二、三的边界
- 要不要保留旧 ingest
- 数据模型是否要升级为通用文档语义

这一步非常关键。对于架构改造型任务，先做设计讨论，通常比一上来写代码更重要。

### 4.2 明确要求 spec 和 plan 沉淀

你要求先形成 spec，再形成实现计划，然后再执行。这是本次协作质量能稳定下来的核心原因之一。

好处很直接：

- 设计决策有书面记录
- review 有明确对象
- 实现阶段不容易来回摇摆
- 事后回顾也更容易

### 4.3 关键决策点始终由你拍板

例如这些点，你都亲自拍板了：

- 选择方案二，而不是单类大 Loader
- 知识模型升级为通用文档语义
- 第一版就做去重与增量更新
- HTML 暂不处理
- 双库一致性按离线场景降级，而不是做在线版本切换

这很符合“人主导、Agent 落地”的最佳实践。

### 4.4 你提供了高价值约束

这次很多关键约束，其实都是你主动补出来的：

- Windows native + PowerShell
- 文件读取统一按 UTF-8 处理
- 后续操作都在 `aiCoach` conda 环境中执行
- 最终希望走 API，而不是本地重模型部署

这些约束一旦明确，Agent 的探索空间会更聚焦，返工会更少。

### 4.5 你提供了真实日志和真实样本

你没有只说“有问题”，而是不断贴出：

- traceback
- Alembic 报错
- Milvus schema 报错
- collection 未加载报错
- 实际 PDF 分块效果截图

这类输入比抽象描述有价值得多。它能显著提高定位效率，也是与 Agent 协作时最值得坚持的习惯之一。

### 4.6 你会对 Agent 的输出做二次判断

你没有盲信我给出的每一步结果，而是会：

- 重新核对 spec
- 引入另一位 agent 的 review findings
- 对“双库一致性”提出场景修正
- 对 MinerU 部署模式提出重新审视

这一步很重要。AI 协作不是“接受输出”，而是“持续校正输出”。

## 5. 这次值得改进的地方

这部分不是问题归因，而是下次可以更省时间的地方。

### 5.1 可以更早明确“运行方式”而不是只说“使用某工具”

这次 MinerU 就是典型例子。我们先后经历了：

- 本地 CLI / pipeline 思路
- 远端 http-client 思路
- `langchain_mineru` 探索
- 最终切到 `mineru-open-sdk`

根因不是技术能力不足，而是“MinerU”这个名字本身覆盖了多种运行形态。

下次如果一开始就明确：

- 本地部署还是云 API
- 是否允许下载大模型
- 是否接受额外服务端
- 是否必须兼容当前机器资源

会少走很多弯路。

### 5.2 可以更早给出“可运行验收清单”

这次我们在实现后期才逐步暴露出这些运行型前置条件：

- `.env`
- Alembic 迁移
- Milvus collection 重建
- MinerU token
- 示例文件目录

下次更稳的做法是，在实现前就列出最小验收清单，例如：

1. 准备测试文件
2. 准备 `.env`
3. 跑迁移
4. 初始化 Milvus
5. 指定使用的 conda 环境
6. 指定成功标准

这样真实联调会更顺。

### 5.3 密钥不要发到对话里

这次你为了推进测试，把 MinerU token 直接发到了会话中。这个动作可以理解，但从安全角度不推荐。

更好的做法是：

- 直接写入 `.env`
- 只告诉 Agent “token 已配置”
- 如需确认，贴变量名而不是值

### 5.4 可以更早定义“哪些问题允许通过重跑恢复”

你后面做得很好，最终明确了：

- 这是离线导入
- 不需要在线版本切换
- 失败后允许清理并重跑恢复一致性

如果这类运行边界能在更前面定下来，spec 的复杂度会更低，设计会更早收敛。

### 5.5 可以提前把“必须长期保留的团队规则”写进仓库

比如这次反复出现的几条规则，其实已经值得进项目常驻文档：

- Windows 下统一 UTF-8
- 优先使用 `aiCoach` 环境
- 导入脚本属于离线维护操作
- 新格式接入应遵守统一 ingest 模型
- 敏感配置放 `.env`

这样下次协作时，Agent 不需要等你再次口头提醒。

## 6. 面向下次协作的推荐流程

结合这次经验，我建议你以后和 Agent 协作编码时，优先使用下面这套流程。

### 6.1 启动阶段

先明确 5 件事：

- 目标
- 作用范围
- 环境约束
- 资源约束
- 成功标准

推荐表达模板：

```text
目标：
当前代码位置：
希望修改的范围：
不希望采用的方案：
运行环境与依赖约束：
验收标准：
```

### 6.2 设计阶段

对于中大型改造，先要求：

- 方案讨论
- spec 文档
- 实现计划

不要跳过这一层直接编码。

### 6.3 执行阶段

在执行中重点要求：

- 每轮修改前说明准备做什么
- 遇到异常时先定位根因，再给修复
- 关键取舍点必须回到人这里确认
- 修改后附带验证步骤

### 6.4 验收阶段

验收至少分三层：

- 单元测试
- 真实样本运行
- 文档与分支收尾

如果涉及数据库、向量库、外部 API，还应额外验证：

- 迁移是否能跑
- 初始化是否幂等
- 删除重建是否可恢复
- 错误提示是否足够清楚

## 7. 适合本项目的长期协作规范

基于本次经验，建议把以下内容逐步固化到项目中：

### 7.1 项目级规则

- 统一编码与文件读写为 UTF-8
- 默认运行环境为 `aiCoach`
- 导入系统按“PG 主存储 + Milvus 轻索引”维护
- 敏感配置只进 `.env`

### 7.2 文档级规则

- 重要改造先有 spec
- 再有 implementation plan
- 完成后补 completion report

### 7.3 运行级规则

- 新增可运行功能时，同时提供最小验收命令
- 涉及外部服务时，明确本地模式和远端模式
- 涉及双存储时，明确恢复方式和管理边界

## 8. 对本次合作的评价

如果从“人与 Agent 协作编码”的角度评价，这次合作是比较高质量的一次。

原因不是因为“没有问题”，而是因为：

- 问题暴露得比较充分
- 决策过程清楚
- 文档沉淀完整
- 你会持续校正方向
- 最终落实到了代码、测试、文档、合并和清理

更重要的是，这次合作已经不只是“让 Agent 写代码”，而是在逐步形成你自己的协作方法论。这一点比单次功能完成更有长期价值。
