# 统一知识导入改造完成报告

## 1. 背景与目标

本次改造围绕项目原有的 `scripts/ingest.py` 展开。旧实现只能处理单一 Markdown 文件，并且导入链路中存在以下局限：

- 仅支持 `md`
- 文档解析、切分、向量化、持久化逻辑耦合在一个脚本中
- 数据模型偏讲义语义，不适合扩展到通用知识文档
- 缺少图片资产追踪与统一 citation 支撑

本次改造的目标是：

- 将导入能力升级为面向通用知识库的统一 ingest 系统
- 第一版支持 `txt / docx / md / pdf`
- PostgreSQL 作为主存储，Milvus 作为向量索引
- 支持 PDF 图片资产追踪，为后续 RAG citation 提供出处与资源关联
- 为后续的去重、增量更新、管理脚本扩展保留结构基础

## 2. 最终方案概览

### 2.1 文档加载体系

本次采用“按类型拆分 loader”的方案：

- `LangChainDocumentLoader`
  - 处理 `txt / docx / md`
- `MinerUPdfLoader`
  - 处理 `pdf`
  - 最终接入 `mineru-open-sdk`

整体流程为：

1. 扫描目录中的支持文件
2. 根据扩展名选择 loader
3. 解析为统一的 `IngestedDocument`
4. 切分为 `IngestedChunk`
5. 向量化
6. 写入 PostgreSQL 与 Milvus

### 2.2 持久化策略

采用：

- PostgreSQL 存储主数据
- Milvus 存储向量索引

其中：

- PG 保存文档、chunk、资产、chunk-asset 关系、ingest job
- Milvus 仅保存：
  - `vector_id`
  - `chunk_id`
  - `document_id`
  - `preview_text`
  - `file_type`
  - `page_idx`
  - `embedding`
  - `metadata`

Milvus 不再保存完整正文，正文以 PostgreSQL 为准。

### 2.3 PDF 处理策略

PDF 最终采用 `mineru-open-sdk`，不再依赖本地 MinerU 重模型部署。

当前策略：

- 默认 `precision`
- 未配置 `MINERU_TOKEN` 时自动降级到 `flash`
- 使用 SDK 返回的：
  - `markdown`
  - `content_list`
  - `images`

图片资产会被保存到稳定目录，并在 `knowledge_assets` 与 `knowledge_chunk_assets` 中建立映射关系。

## 4. 过程中遇到的关键问题与处理

### 4.1 Worktree 环境变量读取失败

问题：

- 在 worktree 中运行脚本时，`.env` 未被正确读取
- 导致 PostgreSQL、Milvus 等配置缺失

处理：

- 在 [app/core/config.py](D:/Code/python/AILearningCoach/.worktrees/codex-unified-knowledge-ingest/app/core/config.py) 中增加回退逻辑
- 优先读取当前 worktree `.env`
- 不存在时回退主仓库根目录 `.env`

### 4.2 Windows 编码问题

问题：

- 在 Windows + PowerShell 环境下执行 Alembic 时出现 `gbk` 解码错误

处理：

- 明确后续测试与运行统一使用 UTF-8 环境变量
- 文档与运行说明中统一强调：
  - `PYTHONUTF8=1`
  - `PYTHONIOENCODING=utf-8`

### 4.3 Milvus schema 演进导致的兼容问题

问题：

- 初期从旧 collection 演进到新 schema 时，出现字段数量不一致
- 后续又出现 `preview_text` 超长

处理：

- 重建 collection
- 实现 UTF-8 字节安全截断
- 最终移除 Milvus 中的完整正文存储，仅保留 `preview_text`

### 4.4 PostgreSQL 提交失败时的半成功状态

问题：

- Milvus 已写入，但 PG 回滚时会留下孤儿向量

处理：

- 在 [app/ai/rag/ingest/persistence.py](D:/Code/python/AILearningCoach/.worktrees/codex-unified-knowledge-ingest/app/ai/rag/ingest/persistence.py) 中加入补偿删除逻辑
- 若 PG 提交失败，则回删本次插入的 Milvus 向量

### 4.5 MinerU 路线多次调整

本次 PDF 路线经历了三次收敛：

1. 本地 CLI / pipeline
2. 远端 `*-http-client`
3. 最终切到 `mineru-open-sdk`

最终原因：

- 本地部署计算资源要求高，不符合当前需求
- `*-http-client` 更像请求自建服务，不是直接面向公有 API 的最优方案
- `mineru-open-sdk` 更贴合“直接调用 MinerU 在线能力”的需求
- 同时还能拿到 `content_list` 和 `images`，便于保留资产追踪设计

### 4.6 PDF 分块过碎

问题：

- 初版把 MinerU 的 `content_list` 每个文本块都当成单独文档
- 导致封面作者、机构、邮箱等信息在 Milvus 中被拆成大量碎片

处理：

- 在 [app/ai/rag/ingest/loaders/mineru_pdf_loader.py](D:/Code/python/AILearningCoach/.worktrees/codex-unified-knowledge-ingest/app/ai/rag/ingest/loaders/mineru_pdf_loader.py) 中增加页级聚合
- 先按页合并文本块，再交给统一 splitter
- 目前已显著减少碎片化问题

### 4.7 Milvus collection 生命周期问题

问题：

- 实测中出现：
  - collection 不存在
  - collection 未 load 就 delete

处理：

- 在 [app/ai/rag/milvus_client.py](D:/Code/python/AILearningCoach/.worktrees/codex-unified-knowledge-ingest/app/ai/rag/milvus_client.py) 中新增幂等初始化逻辑
- `insert()` 前自动确保 collection 存在
- `delete()` 前自动 `load()`

## 5. 当前行为说明

### 5.1 同一路径文件的重建策略

当前导入策略是“按文件路径全量替换”：

- 以 `source_path` 作为文件级主锚点
- 导入同一路径文件时：
  1. 先删除该文件在 Milvus 中的旧向量
  2. 再删除该文件在 PostgreSQL 中的旧文档、chunk、资产与关系
  3. 重新导入和重建

这保证了离线导入场景下的双库一致性。

### 5.2 PDF 当前分块语义

当前 PDF 导入并不是“每页一个 chunk”，而是：

- 先按页聚合为页级文档
- 再用统一 `DocumentSplitter` 进行二次切分

所以最终 chunk 数量会大于页数，但远小于原始 `content_list` 的碎块数量。
