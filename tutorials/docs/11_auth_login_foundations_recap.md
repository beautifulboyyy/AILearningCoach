# 登录认证学习总结

这份总结记录了当前阶段关于注册、登录、密码哈希、JWT、Bearer Token 的核心理解。

这一阶段的目标不是把认证安全体系学到最深，而是先建立一个能读懂项目代码、能解释基本流程、能支撑后续接口学习的最小知识框架。

## 当前阶段学到了什么

### 1. 注册和登录分别解决什么问题

可以把认证过程拆成三件事：

1. 注册：保存用户身份信息
2. 登录：校验用户身份并发放凭证
3. 鉴权：后续请求用凭证证明“当前请求是谁”

也就是说：

注册是“存身份”，登录是“发凭证”，鉴权是“验凭证”。

### 2. 注册流程的基本原理

注册的核心流程是：

1. 前端提交用户名、邮箱、密码
2. 后端检查用户名或邮箱是否已存在
3. 密码不能明文存储，要先做哈希
4. 把用户数据写入数据库
5. 返回注册成功结果

你已经能读懂这里的大部分数据库逻辑：

1. 查重
2. 创建用户对象
3. 保存到数据库
4. 创建关联画像

### 3. 登录流程的基本原理

登录的核心流程是：

1. 前端提交账号和密码
2. 后端按用户名或邮箱查用户
3. 校验输入密码与数据库中的哈希密码是否匹配
4. 如果匹配，生成 token
5. 把 token 返回给前端

这一步里数据库操作你已经基本能看懂，新的重点主要在：

1. 密码哈希校验
2. access token / refresh token
3. 后续请求中的 Bearer 认证

## 密码哈希的第一层理解

注册和登录里最重要的安全基础之一就是：

后端不应该保存明文密码。

所以注册时会做：

1. 用户输入明文密码
2. 后端对密码做哈希
3. 数据库存的是哈希后的结果

登录时不会把数据库里的密码“解密出来”比较，而是：

1. 用户再次输入密码
2. 后端用校验函数判断该密码是否与哈希结果匹配

这一层你可以先记成：

1. 注册时哈希
2. 登录时校验哈希

## JWT 的第一层理解

JWT 可以理解为一种具体的 token 格式。

更准确地说：

1. token 是泛称，表示“凭证”
2. JWT 是其中一种常见的 token 结构

JWT 全称是：

JSON Web Token

你当前只需要掌握它的三个关键特征：

1. 它是一个字符串凭证
2. 它内部带有结构化信息
3. 它可以被后端验证是否被篡改

### JWT 的作用

JWT 的核心作用是：

登录成功后，后端把用户身份信息编码成一个可验证的凭证发给前端。

前端后续访问受保护接口时带上这个凭证，后端就能识别当前用户。

### JWT 的结构

JWT 通常由三部分组成：

1. Header
2. Payload
3. Signature

你当前不用深挖编码细节，只要记住：

1. Payload 里通常会有用户 id、过期时间、token 类型等信息
2. Signature 用于保证 token 没被篡改

### 为什么说 JWT 是“无状态凭证”

因为后端不一定需要在服务器内存里保存“登录会话对象”。

它拿到 token 后，只要：

1. 验签成功
2. 没过期
3. 内容合法

就能知道当前请求是谁。

## access token 和 refresh token

当前项目里使用了两类 token：

### 1. access token

作用：

1. 用于访问需要登录的接口
2. 生命周期较短

可以把它理解成：

临时通行证。

### 2. refresh token

作用：

1. 生命周期更长
2. 当 access token 过期时，用它换一套新的 token

可以把它理解成：

补办临时通行证的长期凭证。

### 为什么要分两种 token

因为只用一个长期 token 风险较高。

拆成 access + refresh 的好处是：

1. access token 泄露时风险窗口较短
2. 用户体验又不会太差

你当前的理解已经比较清晰：

refresh token 的作用，某种意义上就像“轻量级重新登录”。

## `token_type: bearer` 是什么

`token_type: bearer`` 的意义不是说 JWT 内部类型叫 bearer。

它表示的是：

这个 token 要按 Bearer Token 的方式放进 HTTP 请求头使用。

也就是后续请求通常要这样带：

```http
Authorization: Bearer <access_token>
```

这里：

1. `Authorization` 是请求头字段
2. `Bearer` 是认证方案名称
3. 后面跟着真正的 token 字符串

所以 `token_type: bearer` 的作用可以理解成：

告诉前端“你该怎么带这个 token 去访问后端”。
