# RAG 与 Milvus 学习总结

这份总结记录了当前阶段关于 RAG、向量检索、Milvus、混合检索与 rerank 的第一轮系统学习成果。

当前目标不是把每一种高级检索算法都学到最深，而是先建立一个足够稳的知识框架，让后续阅读项目里的 RAG 代码时能做到：

1. 看懂主流程
2. 解释关键技术点
3. 判断当前实现的复杂度与边界

## 当前对 RAG 的整体理解

RAG（Retrieval-Augmented Generation，检索增强生成）的核心目标是：

在大模型生成之前，先从外部知识库中检索相关信息，把这些信息作为上下文提供给模型，从而提升回答的准确性、时效性与可解释性。

可以把 RAG 的价值总结为：

1. 补充模型参数知识之外的信息
2. 缓解知识过时问题
3. 降低幻觉
4. 让回答尽量“有依据”

## RAG 的完整流程框架

当前可以把 RAG 拆成三大阶段：

### 1. 离线索引阶段

主要步骤：

1. 文档加载
2. 文档清洗
3. 文本分块
4. 向量化
5. 存入向量数据库
6. 创建索引

这一阶段的目标是：

把原始文档转换成可检索的知识库。

### 2. 在线检索阶段

主要步骤：

1. 用户输入 query
2. query 预处理或改写
3. query 向量化
4. 在向量库中召回候选文档
5. 可选：结合关键词检索做混合检索
6. 可选：做 rerank 精排

这一阶段的目标是：

找到最适合提供给大模型的上下文片段。

### 3. 在线生成阶段

主要步骤：

1. 格式化检索结果
2. 拼接 prompt
3. 调用 LLM 生成答案
4. 添加来源、引用或置信度

这一阶段的目标是：

让模型“基于检索结果”回答，而不是完全依靠参数记忆回答。

## 当前对 Milvus 的理解

Milvus 是一个专业的向量数据库，主要用于：

1. 存储高维向量
2. 建立向量索引
3. 支持高效近似搜索
4. 配合元数据过滤做更可控的检索

当前阶段可以把 Milvus 理解成：

RAG 系统里的向量存储与向量搜索基础设施。

## Milvus 中几个关键层次的区别

这一块很容易混，现在已经建立了第一层清晰理解。

### 1. Schema

作用：

定义集合中有哪些字段，以及字段的类型和配置。

例如：

1. 主键字段
2. 向量字段
3. 文本字段
4. metadata 字段

可以类比成：

关系型数据库里的表结构定义。

### 2. Collection

作用：

Milvus 中实际存放向量数据的集合。

可以类比成：

关系型数据库中的一张表。

### 3. Index

作用：

为向量搜索建立加速结构。

注意：

索引不是数据本身，而是为了加速搜索而建立的数据组织方式。

### 4. Search Params

作用：

控制查询时如何搜索。

注意：

它不是建表配置，而是查询阶段的参数。

## 当前对向量检索的理解

在最基础的向量检索中，流程可以理解为：

1. 将 query 向量化
2. 与知识库中存储的 document chunk 向量做相似度比较
3. 召回最相关的 top_k 文档

这一步是 dense retrieval，即基于稠密向量的语义检索。

它的特点是：

1. 语义召回能力强
2. 能处理表述不一致但语义相近的问题
3. 精确关键词命中不一定稳定

## 当前对 sparse retrieval / BM25 的理解

sparse retrieval 更偏向传统关键词检索。

常见代表：

1. BM25
2. 稀疏向量检索

它的特点是：

1. 关键词命中更精确
2. 对专有名词、固定术语、精确表达更友好
3. 语义泛化能力不如 dense retrieval

## 当前对 hybrid retrieval 的理解

混合检索的本质是：

把 dense retrieval 的语义召回能力与 sparse retrieval 的关键词匹配能力结合起来。

典型做法有：

1. 各自独立召回，再做融合
2. 先做向量召回，再在候选集上跑 BM25
3. 用 RRF 或加权方式融合排序结果

当前理解可以总结为：

hybrid retrieval 的目标不是替代某一种检索，而是同时兼顾语义相关性与关键词精确度。

## 当前对 rerank 的理解

你已经建立了一个非常关键的认识：

rerank 不是第一次检索，而是对第一次召回的候选结果做第二次更精细的排序。

可以把 rerank 理解成：

二阶段检索中的“精排阶段”。

### rerank 的基本流程

1. 第一阶段：粗召回
2. 第二阶段：精排

第一阶段更关注：

1. 速度
2. 召回率
3. 不要漏掉相关候选

第二阶段更关注：

1. 真正的相关性排序
2. 哪几个候选最适合给大模型

### 对 rerank 的当前理解结论

你现在可以这样记：

先广撒网，再精挑选。

或者更正式地说：

rerank 是对第一阶段召回候选文档的二次精排，而不是重新检索。

## 当前项目中的 rerank 与理论 rerank 的区别

这一点是本轮学习中的一个重要收获。

理论上更强的 rerank 往往会使用：

1. Cross-Encoder
2. 专门的 reranker 模型
3. 更高精度但更慢的判断方式

而当前项目中的 [reranker.py](/D:/Code/python/AILearningCoach/app/ai/rag/reranker.py) 使用的是：

1. 关键词 overlap 加分
2. 标题命中加分
3. 与原始分数融合

所以当前项目是：

有 rerank 思想，但不是重模型精排。

这非常适合面试时区分：

1. 你理解 rerank 的理论意义
2. 你也知道当前项目里的 rerank 是轻量规则版实现

## 为什么 chunking 很重要

这一轮学习中一个非常值得强化的认知是：

RAG 检索的对象不是整个原始文档，而是 chunk。

所以 chunking 会直接影响检索质量。

### chunk 太大时

问题：

1. 噪声多
2. 主题混杂
3. 命中后提供给模型的上下文不够聚焦

### chunk 太小时

问题：

1. 语义不完整
2. 上下文被切断
3. 模型可能拿到的信息不足以回答问题

### overlap 的意义

适当 overlap 可以减少语义截断问题，但 overlap 太大又会带来冗余和重复。

所以当前阶段可以建立一个重要结论：

chunking 往往决定了 RAG 的上限。

## 为什么 query embedding 和 document embedding 必须在同一向量空间

这是向量检索最底层的原则之一。

只有在同一个 embedding 空间里，query 向量和文档向量的相似度比较才有意义。

更准确地说：

1. query 和 document 应该使用兼容的 embedding 模型
2. 这样“相近语义 -> 空间中相近位置”的规则才一致

如果两者不是同一空间，就像使用不同的坐标系做距离比较，结果通常不可靠。

## metric_type 的第一层理解

metric_type 决定“如何判断两个向量像不像”。

当前阶段重点掌握三个概念即可：

### 1. COSINE

关注两个向量的方向是否一致。

常用于：

文本语义检索。

### 2. IP（Inner Product）

看的是向量内积。

如果向量已经归一化，那么：

IP 和 COSINE 往往可以近似看作等价的工程选择。

### 3. L2

即欧氏距离。

表示空间中的直线距离。

### 当前理解结论

教材中看到 COSINE，项目中看到 IP，并不矛盾。

因为：

归一化向量下，IP 和 COSINE 经常会得到相近效果。

## 为什么项目里 Milvus 用 IVF_FLAT，而不是 HNSW

当前阶段不追求把所有索引都学深，只要先掌握一个正确框架。

### IVF_FLAT 的直觉理解

先把大库粗分成很多簇（桶），搜索时先找到最相关的几个桶，再在桶内做搜索。

特点：

1. 思路清晰
2. 实现稳定
3. 易于解释和调参

### HNSW 的直觉理解

通过多层近邻图结构做搜索，查询时沿着图逐步逼近最相似结果。

特点：

1. 精度通常较高
2. 速度也可能很好
3. 但构建和理解复杂度更高

### 当前项目为什么使用 IVF_FLAT

当前项目更偏：

1. 工程教学结合
2. 中等规模数据
3. 稳定与可维护优先

所以选择 IVF_FLAT 是一个合理的工程折中。

结论不是：

IVF_FLAT 更高级。

而是：

它更适合当前项目的复杂度与维护目标。

## top_k、nlist、nprobe 的第一层理解

这三个参数非常容易混。

### 1. top_k

作用：

最终返回多少条结果。

影响：

1. 太小可能漏信息
2. 太大可能带来噪声和上下文污染

### 2. nlist

作用：

在 IVF 索引中，把数据分成多少个桶。

影响：

1. 分桶粒度
2. 索引的组织方式

### 3. nprobe

作用：

查询时实际会探测多少个桶。

影响：

1. 召回率
2. 查询速度

### 当前理解结论

可以直接这样记：

1. nlist 决定怎么分桶
2. nprobe 决定一次查多少桶
3. top_k 决定最终拿多少条结果

## 当前学习中最重要的框架结论

现在你已经不只是会背“RAG 是检索增强生成”，而是建立了一个更工程化的理解：

### RAG 的质量取决于

1. 文档质量
2. chunking 质量
3. 检索质量
4. rerank 质量
5. prompt 质量

### Milvus 的关键职责是

1. 存向量
2. 建索引
3. 高效搜索
4. 配合过滤与检索参数完成召回

### 检索策略的关系可以总结成

1. dense retrieval 负责语义召回
2. sparse retrieval 负责精确匹配
3. hybrid retrieval 负责融合两者优势
4. rerank 负责对候选集进行精排
